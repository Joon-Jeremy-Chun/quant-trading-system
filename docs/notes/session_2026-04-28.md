## 2. Git Push 문제 해결 과정

### 문제 원인
레포 히스토리에 `outputs/`, `figures/` 대용량 파일(앵커 CSV, figure PNG)이 커밋되어 있어서
매 push 시 **2 GiB** 패키지를 전송해야 했음. GitHub HTTP 엔드포인트가 ref 업데이트 단계에서 500/408 에러 반환.

```
Writing objects: 100% (22551/22551), 2.02 GiB   ← 데이터는 전송됨
error: RPC failed; HTTP 500                       ← ref 업데이트 실패
Everything up-to-date                             ← 재시도 시 (객체는 있음)
```

### 브랜치 변천사

```
origin/main (APR28-01)  ← GitHub 기준점
    │
    ├── main (local)
    │   └── APR28-02: iex→sip fix (미push)
    │
    ├── codex-pi-live-tranche-safety  ← 다른 에이전트 작업
    │   ├── APR28-02
    │   ├── Harden Pi pipeline (안전장치 추가)
    │   └── APR28-03: 이메일 날짜 fix
    │
    ├── codex/pi-model-artifact-handoff  ← GitHub push 성공 ✅
    │   ├── Add Pi model artifact handoff pipeline
    │   └── Document Pi model artifact handoff
    │
    └── work/pi-manifest-top20  ← 최종 통합 브랜치 ✅
        ├── [codex/pi-model-artifact-handoff 기반]
        ├── APR28-05: manifest top_n 10→20
        └── APR28-06: QQQ+RKLB 4자산 추가
```

### 해결책: `codex/pi-model-artifact-handoff` 아키텍처 채택

다른 에이전트가 만든 브랜치가 GitHub push 문제의 근본 해결책을 제시:

1. **`models/live/latest_model_manifest.json`** 도입 — 워크스테이션이 모델 설정을 Pi에 전달하는 작은 JSON 파일 하나
2. **`.gitignore` 정리** — `outputs/`, `figures/` 등 대용량 재현 가능 파일 제외
3. **`git rm -r --cached`** — 이미 트래킹된 20,637개 파일 untrack
4. **SSH remote URL 전환** — HTTPS에서 `git@github.com:...`으로 변경 후 push 성공

### 최종 채택 아키텍처

```
워크스테이션 (Windows)
  - 연구, 앵커 최적화, 백테스트
  - 월 1회 signals 생성 → outputs/live/latest_*_signal.json
  - models/live/latest_model_manifest.json 업데이트
  - GitHub push (작은 파일만, 빠름)
        │
        ▼
    GitHub (sync layer)
  - 코드, 시그널 JSON, manifest만 저장
  - outputs/, figures/ 제외 (.gitignore)
        │
        ▼
Raspberry Pi
  - git pull --ff-only
  - GLD/BRK-B: --build-signal (Pi가 앵커 스냅샷 보유)
  - QQQ/RKLB: 기존 GitHub 시그널 사용 (validate_existing_signal)
  - Alpaca paper 주문
  - 이메일 보고서 발송
```

## 4. 이메일 보고서 메커니즘 (상세)

### 전체 흐름

```
Pi 파이프라인 실행 (PT 13:00)
    │
    ├── 1. git pull (최신 시그널/코드 수신)
    ├── 2. 데이터 업데이트 (update_daily_price_data.py)
    ├── 3. 시그널 재계산 (run_objective2_latest_live_signal.py)
    │       GLD/BRK-B만 Pi 로컬 재계산
    │       QQQ/RKLB는 GitHub에서 pull한 기존 시그널 사용
    ├── 4. 멀티에셋 정규화
    │       raw 합계 > 100% → 비율 정규화
    ├── 5. 자산별 주문 (legacy_tranche_order.py)
    │       가격: yfinance 종가 (Alpaca IEX 폴백)
    │       심볼: BRK-B → BRK.B 변환 (Alpaca 형식)
    └── 6. 이메일 발송 (send_live_daily_report.py)
```

### 시그널 → 비중 정규화 로직

```python
raw_weights = {
    "GLD":  1.00,   # BUY strong
    "BRK-B": 0.55,  # BUY moderate
    "QQQ":   0.00,  # HOLD
    "RKLB":  0.57,  # BUY
}
total = 2.12  # > 1.0 → 정규화
final = {
    "GLD":  1.00/2.12 = 47.2%,
    "BRK-B": 0.55/2.12 = 26.0%,
    "QQQ":  0%,
    "RKLB": 0.57/2.12 = 26.9%,
}
```

### 이메일 스크립트 (`send_live_daily_report.py`)

#### ASSETS 리스트 (4자산)
```python
ASSETS = [
    {"symbol": "GLD",   "slug": "gld",  "color": "#c8a020", ...},
    {"symbol": "BRK-B", "slug": "brkb", "color": "#1a5276", ...},
    {"symbol": "QQQ",   "slug": "qqq",  "color": "#1e8449", ...},
    {"symbol": "RKLB",  "slug": "rklb", "color": "#922b21", ...},
]
```

#### 이메일 구성

1. **시그널 로드**: `outputs/live/latest_{slug}_signal.json` (4개)
2. **트랜치 로드**: `outputs/live/{slug}_tranche_order_*.json` (최신 파일)
3. **비중 계산**: 트랜치 weight_override 우선, 없으면 시그널 raw weight 정규화
4. **차트 생성** (`build_chart_png`):
   - 6개월 가격 차트 (matplotlib)
   - 라이브 시작일(asof_date)부터 비중 패널 표시
   - 인라인 이미지로 이메일 첨부 (`Content-ID: chart_{slug}`)
5. **HTML 빌드** (`build_html`): 동적 N자산 지원
   - 포트폴리오 overview (비중 테이블 + 컬러 바)
   - 정규화 배너 (적용 시)
   - 자산별 섹션: 시그널, 가격, 예측수익률, 모델 정보, 주문 내역
6. **제목 생성**: BUY 자산만 표시
   ```
   [GLD BUY 47%] [BRK-B BUY 26%] [RKLB BUY 27%] $432.97 · 2026-04-29 (signal: 2026-04-24)
   ```
   - `date.today()` = 실행일 (시그널 날짜와 구분)
   - HOLD 자산(QQQ)은 제목 제외, 본문에만 표시
